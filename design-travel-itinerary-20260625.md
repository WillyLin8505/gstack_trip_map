# Design: 旅遊行程排序與天數規劃工具

Generated by /office-hours on 2026-06-25
Branch: main
Repo: gstack_food_map
Status: APPROVED
Mode: Startup

## Problem Statement

用戶想規劃旅遊行程（景點 + 餐廳），但不知道如何安排最順路的順序、幾天最合理，以及需要掌握各地點的真實資訊（營業時間、開放時間、門票）。現有工具（Google Maps）只能手動逐一查詢、手動排序，整個過程花費好幾小時。

核心問題是：**「我已經知道要去哪裡，但我不知道怎麼排才最順路、幾天最好。」**

## Demand Evidence

- 創辦人本人是第一個用戶：「每次規劃行程都很痛苦」
- 現有工作流程：Google Maps + 手動排序，每次花「好幾小時」
- 市場驗證：2026 年行程規劃工具爆炸，但沒有任何工具專注在「用戶已有清單 → 幫我排序」這個問題。現有 AI 工具（Wanderlog、MindTrip）都在解決「幫我發現要去哪裡」，不是「幫我排我已知的地方」
- **需求缺口**：Wanderlog 雖然有地圖拖拉排序，但仍需要手動操作；沒有工具能自動根據開放時間約束做排程優化

⚠️ **尚未驗證的需求**：目前只有創辦人自身的痛點，還沒有第三方用戶確認此痛點。

## Status Quo

用戶當前工作流程：
1. 在 Google Maps 手動標記想去的景點/餐廳（花 30-60 分鐘）
2. 查詢每個地點的營業時間（逐一 Google，花 60+ 分鐘）
3. 嘗試手動分組成天數（需反覆比對地圖位置，花 60+ 分鐘）
4. 排路線順序，確認不繞路（再花 30-60 分鐘）
5. 總計：規劃一趟 3-5 天行程，花費 3-5 小時

真正的競爭對手：**用戶自己拼湊的 Google Maps + 手動排序流程**，不是其他 App。

## Target User & Narrowest Wedge

**目標用戶（修訂後）：**
- **A 類（清單派）**：已有明確景點/餐廳清單，只需要排序和天數建議
- **B 類（目的地派）**：知道要去一個城市，但清單不完整，需要補充推薦 + 排序

⚠️ **已確認的缺口**：創辦人在 Premise 1 提出異議，表示用戶不一定有明確清單。因此工具需同時服務 A 和 B 兩類入口。

**最小可行切入點（Narrowest Wedge）：**
**v1 只支援 A 類用戶（已有清單的人）。B 類用戶（需補充推薦）明確延後到 v2，v1 不做任何推薦功能。**

- 輸入：Textarea，每行一個地點名稱（中英文皆可），上限 30 個地點
- 輸出：地圖視覺化 + 建議天數 + 每天順路排序 + 各地點基本資訊卡（營業時間、票價）

這個版本可以在 2-4 週內上線，可以開始驗證真實需求。

## Constraints

- 無現有用戶或付費客戶（概念階段）
- 資料品質高度依賴第三方 API（Google Places、OpenStreetMap）
- 路線優化演算法複雜度（TSP 問題是 NP-hard，需要近似解）
- 實際開放時間資料可能過時或錯誤（這是業界通病）

## Premises

1. **（修訂）** 用戶可能從兩種入口進來：(A) 已有清單要排序，(B) 有目的地但清單不完整需補充推薦。**v1 只支援 A 類；B 類是 v2 目標。**
2. 最大痛點是「路線順序 + 天數估算」，而不是「景點發現」
3. 商店/景點的真實資訊（營業時間、門票）是核心功能，不是附加功能
4. 這個工具聚焦在旅遊目的地規劃，不是日常交通路線優化

## Approaches Considered

### Approach A: 最小可行版（MVP）— 清單 → 地圖 → 順路排序

核心功能：貼入地點清單，系統自動抓 Google Places 資料，按地理叢集排出建議天數和順序，顯示基本營業時間。

- **Effort**: S（2-4 週）
- **Risk**: Low
- **Pros**: 解決最核心痛點、技術門檻低、可快速驗證需求、無 AI 複雜度
- **Cons**: 無法服務 B 類用戶（需補充推薦的人）、資料更新依賴外部 API

### Approach B: 完整產品版 — AI 推薦 + 約束優化排程

用戶輸入目的地 + 天數 + 偏好，AI 推薦候選清單，用戶勾選後系統用 TSP 演算法 + 真實開放時間約束生成逐天逐小時行程表。

- **Effort**: XL（3-6 個月）
- **Risk**: High（API 成本、資料品質、演算法複雜度）
- **Pros**: 真正解決 A + B 兩類用戶、最高差異化、最完整產品體驗
- **Cons**: 太複雜、在驗證需求前過度投入、未知哪個功能最重要

### Approach C（推薦）: 聯合策略 — MVP 先驗證，再擴展

從 Approach A 的技術簡單性出發，但加入 UI 打磨、可分享連結、以及針對台灣旅遊場景的 onboarding 流程。先以 Google Places API 快速做出可用的 MVP，上線後看哪類用戶最活躍、哪個功能最需要，再決定是否往 B 類用戶和 AI 推薦擴展。

A→C 的額外 3-6 週花在：Supabase 持久化層、行程分享連結（UUID URL）、UI 精修、台灣地區測試資料驗證。**PostHog 分析埋點延後到有外部用戶之前（v1.1），不在核心 MVP 範圍。**

- **Effort**: M（6-10 週，含打磨）
- **Risk**: Med
- **Pros**: 快速驗證、聚焦、技術可行、可在找到真實用戶後再投入大架構
- **Cons**: 需要紀律不要早期過度擴展

## Recommended Approach

**Approach C（聯合策略）：MVP 先驗證，再擴展。**

理由：在任何人確認願意使用（或付費）之前，最貴的事是建一個別人不用的完整產品。先做最小版本，讓真實用戶行為告訴你哪裡最痛。

## Data Model

```
Place {
  id: string           // Google place_id
  name: string
  lat: number
  lng: number
  category: enum       // MUSEUM | RESTAURANT | SCENIC_SPOT | ATTRACTION | OTHER
  opening_hours: OpeningHours[]
  price_level: 0 | 1 | 2 | 3 | 4 | null  // from Google Places API; 0=Free, 1=$, 2=$$, 3=$$$, 4=$$$$
  dwell_time_minutes: number  // system default, user-overridable
}

Itinerary {
  days: Day[]
  total_places: number
  generated_at: timestamp
  share_token: string   // UUID v4, read-only URL, no login, 30-day TTL
  user_id: UUID | null  // NULL in v1 (anonymous); populated in Phase 4 when auth added
}

Day {
  index: number
  places: Place[]      // ordered by greedy nearest-neighbor within cluster
  date: date | null    // user can assign actual dates optionally
}
```

**停留時間預設值（系統判斷，用戶可點擊 inline 調整）：**
- `MUSEUM`（博物館、美術館）：120 分鐘
- `RESTAURANT`（餐廳、咖啡廳）：60 分鐘
- `SCENIC_SPOT`（景點、公園）：90 分鐘
- `ATTRACTION`（廟宇、景區、夜市）：45 分鐘
- `OTHER`：60 分鐘

**分享連結流程（Share Link）：**
1. 用戶點擊「產生行程」→ 後端產生 `share_token`（UUID v4），儲存至 DB
2. 系統顯示可複製的分享連結：`https://app.domain/trip/{share_token}`
3. 分享頁面為 read-only（無法編輯），不需登入，30 天後連結失效
4. 失效連結返回 404 頁面，顯示「此行程連結已過期」

**持久化層（Persistence）：**
- 資料庫：Supabase Postgres — 唯一持久化層（儲存 itineraries, places cache, share_token）
- Place 快取：Supabase `places` table（`SELECT WHERE id = ANY($1)`，batch lookup），TTL 24h 由 pg_cron 清理
- Share token TTL 清理：Supabase pg_cron 或每次查詢時檢查 `expires_at` 欄位

## Algorithm Decisions (MVP)

**叢集演算法**：k-means on lat/lng（Haversine distance，非 Euclidean，避免高緯度失真）
- k（建議天數）= ceil(total_dwell_time_hours / 8)，下限 1，上限 7 天
- 每日可用遊覽時間預設為 8 小時
- 邊界情形：少於 3 個地點 → k=1（單日行程，不做叢集，直接排序）
- k-means 的距離失真在台灣規模（23-25°N，同城市內）可接受（誤差 <2%）；若用戶輸入跨區（台北到墾丁）顯示警告

**排序演算法**：每日叢集內用 **Greedy Nearest-Neighbor Heuristic**
- 起點：叢集地理中心最近的地點（MVP 不支援指定起點）
- 足夠應對 30 個地點以內的清單；時間複雜度 O(n²)，即時完成
- 用戶可手動拖拉調整順序（調整不觸發重新排序，只做 UI 更新）

**地圖套件**：Google Maps JS API（與 Google Places API 同源，減少整合複雜度）

**地理範圍限制**：若地點跨度超過 200 km 顯示警告，不封鎖。

**v1 不含開放時間約束排程**：v1 只顯示開放時間資訊卡，不主動調整排序以符合開放時間。開放時間約束排程是 v2 的核心功能。v1 的核心差異化是「路線叢集排序 + 天數建議」。

**停留時間手動調整**：用戶可點擊資訊卡 inline 修改停留時間 → 觸發天數重新計算（k 重算），重新排序。

## Error Handling & Degraded State

- **Places API 找不到地點**：顯示地點名稱 + 警告標記「資訊待補充」，允許用戶手動填入時間/票價
- **API 403 / 401（key 無效或超出配額硬上限）**：server 返回 500 + 用戶訊息「系統配置錯誤，請稍後再試」；後端 `console.error` 記錄（v2 接 Sentry alert）
- **API quota 耗盡（429 rate limit）**：顯示系統通知，該地點 fallback 到 OpenStreetMap Nominatim（座標查詢）
- **資料過時**：在資訊卡上顯示「資料來自 Google，請出發前確認」免責聲明
- **雙重送出（duplicate submit）**：送出按鈕在請求進行中 disabled；server side 以 place_list hash 作為 idempotency key，重複請求返回正在進行的結果
- **XSS 防護**：所有來自 Google Places API 的字串（name, address, 等）一律透過 React JSX 渲染，禁止使用 `dangerouslySetInnerHTML`

## Open Questions

1. **資料來源**：Google Places API 或 OpenStreetMap 哪個更適合？（前者資料品質高但有費用；後者免費但台灣資料不完整）— **MVP 用 Google Places，加快開發速度**
2. **B 類用戶（需補充推薦）的優先順序**：**明確決定：B 類用戶（景點發現 + 推薦）v1 不支援，v2 依用戶回饋再評估**
3. **付費模型**：免費試用後收費？等 v1 有用戶後再決定。B2B 旅遊社方向留待消費者端驗證後再評估。

## Success Criteria

- 第一版（4 週內）：自己能用來規劃一趟旅程，比 Google Maps + 手動排序省至少 2 小時
- 第一個外部用戶（8 週內）：找 5 個自由行旅客試用，至少 3 人表示「這比我現在的方法好」
- 收費驗證（3 個月內）：至少 1 個用戶願意付費（即使是小額）

## Dependencies

- **Google Places API（New）**（Details + Geocoding）
  - 費用：按 field mask 計費，opening_hours（Contact Data）+ geometry（Basic Data）+ price_level（Atmosphere Data）合計約 $0.03-0.05/place
  - 保守估算：$0.05 × 10 places × 100 sessions/day = $50/day（無快取）
  - 加快取後（同地點 24h 內不重複查詢）：估算降至 $5-10/day MVP 規模
  - 規模化風險：10,000 places/day 無快取 → $500/day；快取後約 $50/day
- **Google Maps JS API**：地圖顯示（與 Places API 同帳號，管理單純）
- **路線排序**：Greedy Nearest-Neighbor（自行實作，O(n²)，不需 Distance Matrix API）
- **Supabase Postgres**：儲存 Itinerary、Place 快取、share_token
- **Upstash Redis**：Vercel serverless 快取（in-memory 不適用），key: `place:{place_id}`, TTL: 24h

## Reviewer Concerns (需在工程設計階段解決)

以下問題在 3 輪對抗性審查後仍未解決，留待工程師在實作前確認：

1. **Tech Stack 未完整定義**：Next.js App Router vs Pages Router？優化演算法跑在 API Route 還是 Edge Function？
2. **DB Schema 缺 DDL**：`Place` 獨立 table 或嵌入 Itinerary JSONB？Index 欄位未定？
3. **k-means 初始化策略**：需指定 k-means++ + fixed seed 確保確定性結果，避免非確定性輸出
4. **拖拉排序套件未指定**：`dnd-kit` 或 `react-beautiful-dnd`？重排後是否持久化至 DB？
5. **畫面流程**：輸入頁 → 結果頁 → 分享頁，還是 single-page flow？影響路由架構
6. **k-means degenerate case**：若地點全部集中在同一叢集，需要 fallback 策略

**CEO-F3 (HIGH) — XSS: place names from Google Places are untrusted input**
Place names should never be inserted via `dangerouslySetInnerHTML`. Next.js JSX escapes by default — this is a "don't violate" rule, not a "build" task.
Auto-decided (P1): Add to plan — explicitly prohibit `dangerouslySetInnerHTML` for any API-sourced string. React default escaping is sufficient.

**CEO-F4 (MEDIUM) — Double-submit prevention missing**
User clicks "排程最佳路線" twice → two concurrent optimization calls, two itineraries created.
Auto-decided (P5): Disable submit button while request is in-flight (UI); server-side: idempotency key (place_list hash) to detect and return the in-progress request.

When auth is added in Phase 4, linking itineraries to users requires an ALTER TABLE on live data. Design `user_id UUID NULL REFERENCES users(id)` into the schema from day 1.
Auto-decided (P2): Add `user_id UUID NULL` to the itineraries table schema. One column now prevents a painful migration later.
