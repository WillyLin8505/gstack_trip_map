# Design: Real Google Maps Routing + Place Info Panel Enhancement

## Problem Statement

The travel itinerary optimizer displays travel times between stops using a Haversine
distance approximation (× 15 min/km). This often diverges significantly from real
Google Maps travel times, especially in dense urban areas or where roads curve. Users
notice the discrepancy instantly when they compare against their own Maps app.

Additionally, the left-hand PlaceCard panel shows minimal information — only a category
badge and a "closes soon" boolean warning. Users have no way to know whether a place
is actually open when they're planning to arrive, or to read a brief description to
confirm it's worth visiting.

Four concrete changes needed:
1. Replace displayed travel times with real Google Maps Directions API times
2. Show today's actual open/close hours in each PlaceCard
3. Show a brief editorial description in each PlaceCard
4. Remove price_level display (user confirmed: 不需要提供票價)

## What Makes This Cool

Right now the app does real AI work — k-means++ clustering, greedy nearest-neighbor,
2-opt — but the travel times it shows are fake. After this change, every number in
the itinerary comes from a real Google Maps call. The app stops feeling like a
prototype and starts feeling authoritative. The "14 min walk" will match what users
see when they actually tap directions in Maps.

## Constraints

- Must NOT change the optimizer algorithm (too expensive: N² Directions API calls)
- Directions API called after optimization, once per consecutive pair per day
- editorialSummary is not always populated — component must handle null gracefully
- weekdayDescriptions returns 7 strings (Mon–Sun, localized) — pick correct index
- API key is already in .env.local as GOOGLE_PLACES_API_KEY
- Must use Google Directions API (not Routes API) to stay on the same GCP key
- Budget: existing Places API usage + N-1 Directions calls per day (small cost)
- TypeScript strict mode — all new fields need types; no `any`

## Premises

1. Haversine is a display problem, not an optimizer problem.
   The optimizer path order is already good enough (Haversine is correct enough for
   geographic clustering). The user's complaint is that displayed numbers mismatch
   real Maps. Fix the display, not the algorithm.

2. Opening hours are already fetched — just not surfaced.
   `regularOpeningHours.periods` is returned by the Places API call in places.ts.
   The component currently converts it to a boolean (open_now). Surfacing today's
   text hours is purely a PlaceCard change — zero additional API cost.

3. editorialSummary is the correct description source.
   Google Places API (New) returns `editorialSummary.text` — a 1-2 sentence neutral
   description written by Google. It's the right summary for "what is this place"
   without needing another API.

4. price_level is noise without context.
   Without menu prices or ticket context, a 1-4 symbol rating is ambiguous and
   occupies space better used for hours + description.

## Approaches Considered

### Approach A: Display-only real routing (chosen)
After the optimizer computes the ordered visit sequence, call Google Directions API
once per consecutive place pair to get real walking/driving travel time. Replace the
Haversine-estimated travelMinutes in the schedule with real minutes. The optimizer
itself (nearest-neighbor, 2-opt) keeps using Haversine — too expensive to replace.

Effort: M (4 files: types, places.ts, optimizer.ts, PlaceCard.tsx)
Risk: Low — optimizer logic unchanged
Cost: (N-1) per day × num_days Directions API calls ≈ tiny vs. Places calls
Pros: Accurate numbers the user can trust; optimizer stays fast and free
Cons: Possible small delta between optimization path and displayed path (rare)

### Approach B: Full real-routing optimization
Replace Haversine entirely — use Routes API for the distance matrix in nearest-neighbor.
For 30 places: 30² = 900 API calls × $0.005 = $4.50 per user request.
Rejected: too expensive, too slow (30–60s latency).

### Approach C: Opening hours + description only (no routing change)
Skip Directions API. Only add hours text, description, remove price_level.
Rejected: user explicitly asked for real Google Maps routing as the primary request.

### Step 3: Add directionsTime() in places.ts (server-only)
New async function — **must only be called from server-side route handlers,
never imported into client components, to keep GOOGLE_PLACES_API_KEY server-only.
Add `import 'server-only'` at the top of the file (the `server-only` npm package
causes a build-time error if any client component ever imports this module).
Do NOT use `"use server"` — that directive is for Next.js Server Actions only.**
```ts
async function directionsTime(
  originLat: number, originLng: number,
  destLat: number, destLng: number,
  mode: "walking" | "driving"
): Promise<number>
```
Calls `https://maps.googleapis.com/maps/api/directions/json` with:
- origin/destination lat,lng
- mode=walking (default for city itineraries)
- key=GOOGLE_PLACES_API_KEY
- timeout: 3 seconds via `Promise.race([fetch(...), timeout(3000)])`
Returns `routes[0].legs[0].duration.value` (seconds) converted to minutes.
Fallback: return haversine estimate if API call fails or times out.
Typical Directions API latency is 300–800ms. At p-limit(5) and worst-case 9 pairs × 3 days
(27 calls), expected wall-clock overhead is ceil(27/5) × 800ms ≈ 4.3s — within the 10s budget.

### Step 4: Post-optimize travel time refinement in optimizer.ts
After `buildSchedule()` assigns times, add a new async function
`refineTravelTimes(days: Day[], mode: string): Promise<Day[]>` that:
- For each day, iterates consecutive scheduled visits
- Calls directionsTime() for each pair in parallel (p-limit capped at 5)
- Overwrites each visit's `travelMinutes` with the real Directions API value
- Recalculates downstream start times using the same formula as `buildSchedule()`:
  `nextStart = currentStart + durationMinutes + travelMinutes` (walk forward from
  index 1 to end of day array; index 0 has no incoming travel)
- Wraps in try/catch: on any failure, return the pre-refinement schedule unchanged
  and set a response header `X-Travel-Times: estimated` so the client can show a
  disclaimer if desired. Cap concurrency at 5 (stays under GCP's 50 QPS quota while
  supporting multi-day batches).

Call this refinement in the `/api/optimize/route.ts` handler after the sync optimizer
runs but before returning the response.

### Step 5: Update PlaceCard.tsx
Remove price_level / PRICE_LABELS display.
Add opening hours text row (e.g. "Opens 9:00 AM · Closes 9:00 PM").
Add description paragraph below the header (muted color, 2 lines max, no italic — CJK italic renders poorly).
Opening hours: use pre-computed `opening_hours_text` from the Place data.

### Step 6: Handle today's hours text in places.ts
`weekdayDescriptions` from Google Places API (New) is ordered Monday=0…Sunday=6.
`new Date().getDay()` returns Sunday=0…Saturday=6. Use this mapping:
```ts
const dayIndex = (new Date().getDay() + 6) % 7;
const todayText = weekdayDescriptions[dayIndex]; // e.g. "Monday: 9:00 AM – 9:00 PM"
```
**Timezone note (v1 accepted limitation):** `new Date()` uses the server's local timezone,
not the destination's. For an itinerary to Tokyo viewed from New York, "today" may be
off by one day. V1 accepts this limitation. V2 fix: use
`regularOpeningHours.utcOffsetMinutes` (available in the Places API response) to compute
the destination's current day offset before indexing.

Display the full string as-is — it is already human-readable and localized by Google.
No further string parsing needed. Strip the day-name prefix for display:
```ts
const hoursDisplay = todayText?.replace(/^[^:]+:\s*/, "") ?? null;
// e.g. "9:00 AM – 9:00 PM"
```
Store as `opening_hours_text: string | null` on the Place object.

```ts
async function directionsTime(
  lat1: number, lng1: number,
  lat2: number, lng2: number,
  mode = "walking"
): Promise<{ minutes: number; estimated: boolean }> {
  // N4: Gulf of Guinea guard (default 0,0 coords)
  if (Math.abs(lat1) < 0.01 && Math.abs(lng1) < 0.01) {
    return { minutes: travelMinutes(lat1, lng1, lat2, lng2), estimated: true };
  }
  // N11: AbortController — cancels the underlying fetch (not just Promise.race)
  const controller = new AbortController();
  const timer = setTimeout(() => controller.abort(), 3000);
  try {
    // URLSearchParams for safe encoding (Codex: avoid raw interpolation)
    const params = new URLSearchParams({
      origin: `${lat1},${lng1}`,
      destination: `${lat2},${lng2}`,
      mode,
      key: process.env.GOOGLE_PLACES_API_KEY ?? '',
    });
    const url = `https://maps.googleapis.com/maps/api/directions/json?${params}`;
    const res = await fetch(url, { signal: controller.signal });
    // C4 (Eng): clearTimeout moved to finally — body parse must also be covered by timeout
    const data = await res.json();
    // Distinct logging per status (Codex: ZERO_RESULTS/OVER_QUERY_LIMIT also need visibility)
    if (data.status !== 'OK') {
      if (data.status === 'REQUEST_DENIED') {
        console.warn('[Directions API] REQUEST_DENIED — check Directions API is enabled in GCP Console');
      } else if (data.status === 'OVER_QUERY_LIMIT') {
        console.warn('[Directions API] OVER_QUERY_LIMIT — quota exhausted, falling back to travelMinutes');
      } else if (data.status === 'ZERO_RESULTS') {
        console.warn('[Directions API] ZERO_RESULTS — no walking route found, falling back to travelMinutes');
      } else {
        console.warn(`[Directions API] ${data.status} — falling back to travelMinutes`);
      }
      return { minutes: travelMinutes(lat1, lng1, lat2, lng2), estimated: true };
    }
    if (!data.routes?.[0]) {
      return { minutes: travelMinutes(lat1, lng1, lat2, lng2), estimated: true };
    }
    // Codex: clamp to min 1 minute (Math.round can produce 0 for very short legs)
    return { minutes: Math.max(1, Math.round(data.routes[0].legs[0].duration.value / 60)), estimated: false };
  } catch {
    // C1 (Eng): use travelMinutes() from @/lib/utils — haversineKm is in utils.ts not places.ts
    return { minutes: travelMinutes(lat1, lng1, lat2, lng2), estimated: true };
  } finally {
    clearTimeout(timer);
  }
}
```

Export `refineTravelTimes()` function:
```ts
export async function refineTravelTimes(
  days: Day[], mode = "walking"
): Promise<{ days: Day[]; isEstimated: boolean }> {
  try {
    let anyEstimated = false;
    const updatedDays = await Promise.all(days.map(async (day) => {
      if (day.visits.length < 2) return day;
      const results = await Promise.all(
        day.visits.slice(0, -1).map((_, i) =>
          limit(() => directionsTime(
            day.visits[i].place.lat, day.visits[i].place.lng,
            day.visits[i + 1].place.lat, day.visits[i + 1].place.lng,
            mode
          ))
        )
      );
      const newVisits = [...day.visits];
      for (let i = 0; i < results.length; i++) {
        const { minutes, estimated } = results[i];
        if (estimated) anyEstimated = true;
        newVisits[i + 1] = { ...newVisits[i + 1], travel_minutes_from_prev: minutes };
      }
      // Recalculate forward from visit[1] (N1: inline opening_warning re-check)
      for (let i = 1; i < newVisits.length; i++) {
        const prev = newVisits[i - 1];
        const arrival = addMinutes(prev.departure_time, newVisits[i].travel_minutes_from_prev + TRAVEL_BUFFER_MINUTES);
        const departure = addMinutes(arrival, newVisits[i].place.dwell_minutes);
        newVisits[i] = {
          ...newVisits[i],
          arrival_time: arrival,
          departure_time: departure,
          opening_warning: newVisits[i].place.opening_hours?.open_now === false,
        };
      }
      return {
        ...day,
        visits: newVisits,
        total_travel_minutes: newVisits.slice(1).reduce((s, v) => s + v.travel_minutes_from_prev, 0),
      };
    }));
    return { days: updatedDays, isEstimated: anyEstimated };
  } catch {
    return { days, isEstimated: true };
  }
}
```

### Failure Modes

| Failure | User sees | Handler |
|---------|-----------|---------|
| Directions API REQUEST_DENIED | `~` on all times | console.warn + Haversine |
| directionsTime timeout (>3s) | `~` on affected times | AbortController catch |
| refineTravelTimes outer catch | `~` on all times | returns original days |
| Vercel function >30s (extreme) | 502 Gateway | vercel.json maxDuration=30 |
| weekday_descriptions absent | No hours shown | `?? null` guard |
| editorialSummary absent | No description shown | `?.text` optional chain |
| start_date is null | Today's weekday hours | D5 explicit guard |
| Gulf of Guinea lat:0,lng:0 | `~` on affected pair | N4 guard → Haversine |
| places.length > MAX_PLACES_PER_REQUEST | 400 Bad Request | D8 cap |
